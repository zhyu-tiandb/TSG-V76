## 3. 背景与迁移目标

### 3.1 项目背景

TSG 7.x 当前对外服务引擎基于 Java 实现，承担 SSL/TLS 终结、反向代理、认证鉴权、单点登录、NC 模式隧道、统一门户等核心业务流量处理职责。随着客户对高并发、低延迟、轻量部署与稳定性的要求持续提升，现有 Java 引擎主要存在以下瓶颈：

- 高并发 SSL 场景下 JVM GC 压力较大，响应延迟不稳定
- JVM 内存占用高，同等硬件条件下有效吞吐受限
- SSL 握手、HTTP 解析等 I/O 密集型处理在 Java 层性能弱于原生实现
- 部署与调优复杂度较高，运维门槛偏高

OpenResty 基于 Nginx + LuaJIT，具备高并发、低资源占用、原生 SSL/TLS 与代理能力，适合作为 TrustMore 安全网关的新一代对外服务引擎。

### 3.2 迁移目标

| 目标 | 说明 | 量化标准 |
| --- | --- | --- |
| 功能等价 | 所有现有能力在 OpenResty 引擎下完整可用 | 0 功能缺失，白皮书附录功能逐项通过 |
| 性能提升 | 提升并发处理能力与吞吐能力 | 并发连接数不低于 Java 基线，SSL 握手延迟降低至少 30% |
| 行为兼容 | 用户侧、管理侧无感知变化 | 配置可导入，管理操作方式不变，客户端无需升级 |
| 国密兼容 | 完整支持国密协议与算法体系 | SM2/SM3/SM4 与国密 SSL/TLS 能力完整保留 |
| 部署兼容 | 保持原硬件与部署方式适配能力 | SG-5200/5300/5600/6000/8000 持续支持 |
| 运维优化 | 降低资源占用并提高维护效率 | 内存占用降至 Java 版本 70% 以内，支持热加载 |

### 3.3 非目标

- 不新增产品功能
- 不改变客户端交互协议与使用方式
- 不改变管理页面 UI 与管理流程
- 不改变现有硬件平台与密码硬件调用方式
- 不主动改变主路/旁路、反向代理/NC 等既有部署模式

## 4. 迁移范围与总体原则

### 4.1 总体原则

- 仅替换对外服务引擎，管理面、后端数据、客户端逻辑原则上保持不变
- 优先复用现有后端认证、授权、配置与数据能力
- 对高风险能力先 PoC 后实施，避免全量重构后再暴露阻塞问题
- 允许对极少数 OpenResty 不适合承载的能力保留最小辅助进程，但必须单独评审

### 4.2 改造范围

| 模块 | 改造说明 |
| --- | --- |
| HTTP/HTTPS 协议处理 | 替换为 OpenResty http/ssl 能力，保留国际与国密协议支持 |
| 反向代理模式 | 使用 Nginx 反向代理能力配合 Lua 实现路由、重写、注入等逻辑 |
| NC 模式 | 使用 stream 模块配合 Lua 实现隧道建立、策略校验与数据转发 |
| 用户认证鉴权 | 将接入面认证、会话、访问控制逻辑迁移至 Lua 与后端 API 协同 |
| 单点登录 | 使用 Lua 实现表单解析、注入、证书透传、Cookie 处理 |
| 统一门户 | 由静态资源 + Lua/API 方式替代 Java 动态页面承载 |
| 日志审计与监控 | 在 OpenResty 侧重建日志采集点、运行状态统计与对接能力 |

### 4.3 不变范围

- 管理后台主要业务逻辑
- 用户、角色、应用、终端等后端数据模型
- 现有客户端插件与终端交互协议
- 现有硬件适配层与密码设备接口
- 集中管控平台总体纳管方式

## 5. 模块迁移影响分析

### 5.1 影响分类说明

| 分类 | 标记 | 含义 |
| --- | --- | --- |
| 引擎迁移 | 🔄 | 核心逻辑从 Java 迁移至 OpenResty/Lua |
| 适配调整 | ⚠️ | 逻辑基本保留，但需要适配验证或局部调整 |
| 无变化 | ➡️ | 不涉及引擎层重构，后端、管理面或客户端保持不变 |

### 5.2 模块迁移矩阵

#### 5.2.1 链路安全

| 能力 | 当前实现 | 迁移后实现 | 影响类型 | 迁移要点 |
| --- | --- | --- | --- | --- |
| SSL/TLS 加密通道 | Java SSLContext | OpenResty `ngx.ssl` / OpenSSL | 🔄 | 协议处理下沉为 OpenResty 原生能力 |
| 国密 SSL 协议 | Java 国密 SDK | OpenResty + 国密 OpenSSL / Tongsuo | 🔄 | 必须验证与 OpenResty 的集成兼容性 |
| SM2/SM3/SM4 算法 | Java 国密 SDK | 国密 OpenSSL / Tongsuo | 🔄 | 关键依赖项，决定整体迁移可行性 |
| 双网隔离 | 服务端策略下发 | 保持不变 | ➡️ | 客户端行为与隔离策略不因引擎替换而改变 |

## 6. 功能等价与关键约束

### 6.1 必须完整保留的能力

- 数据传输加密能力，包括国际 SSL 与国密 SSL
- 数字证书认证、用户名口令、双因子及外部认证联动
- RBAC、动态角色匹配、应用访问控制
- 终端信息提取、安全检查、终端绑定、零痕迹访问
- 反向代理、NC 模式、单点登录、统一门户
- 双机热备、第三方负载均衡、审计日志、实时监控
- 三权分立、管理员安全策略、集中管控纳管能力

### 6.2 高风险功能保留要求

| 功能点 | 保留要求 |
| --- | --- |
| 国密 SSL（GM/T 0024） | 必须支持 SM2/SM3/SM4 及既有证书体系 |
| CRL/OCSP 证书校验 | 保持与 Java 版本一致的校验效果与异常处理逻辑 |
| 动态端口类应用 | 若 OpenResty 原生支持不足，允许通过辅助进程补齐，但需评审 |
| 用户终端绑定 | 多对多绑定判定规则不得改变 |
| 零痕迹访问 | 保持客户端清理流程与体验一致 |
| NC 模式隧道 | 必须与现有客户端完全兼容，不引入客户端升级要求 |

### 6.3 特殊约束

- 不得引入新的第三方商业软件授权依赖
- 原有国密硬件、USB Key、加密卡调用方式不得改变
- 配置迁移必须可落地，至少提供半自动到全自动的转换方案
- 对外接口口径、日志口径、监控口径必须与既有平台兼容

## 7. 性能与非功能性需求

### 7.1 性能基线与目标

说明：现有稿件中存在两套硬件参考口径。为避免冲突，统一文档保留“基线不低于现网、目标优于 Java”的原则，同时保留两类参考数据供评审选择最终验收口径。

#### 7.1.1 参考基线 A（SG-5300）

| 指标 | Java 基线 | OpenResty 目标 |
| --- | --- | --- |
| 国密最大新建连接数 | 3,000 次/秒 | ≥ 4,500 次/秒 |
| 国密 SSL 事务处理速率 | 5,000 次/秒 | ≥ 8,000 次/秒 |
| 国密最大并发连接数 | ≤ 30,000 | ≥ 50,000 |
| 国密加密带宽吞吐量 | 850 Mbps | ≥ 1,200 Mbps |
| 接入用户数 | ≤ 10,000 | ≥ 15,000 |

#### 7.1.2 参考基线 B（SG-6000）

| 指标 | Java 基线 | OpenResty 目标 |
| --- | --- | --- |
| 国密新建连接数 | 4,500 次/秒 | ≥ 4,500 次/秒 |
| 国密 SSL 事务处理速率 | 8,000 次/秒 | ≥ 8,000 次/秒 |
| 国密最大并发连接数 | 100,000 | ≥ 100,000 |
| 国密加密带宽 | 900 Mbps | ≥ 900 Mbps |
| HTTP 请求处理延迟 P99 | 现网基线 | 不劣于现网 120% |
| SSL 握手延迟 P99 | 现网基线 | 不劣化，目标优于现网 |

### 7.2 统一非功能目标

| 项目 | 要求 |
| --- | --- |
| 内存占用 | 空闲时不高于 Java 版本 70% |
| CPU 使用率 | 同等压力下不高于 Java 版本 |
| 磁盘 I/O | 日志写入开销不高于现网 |
| 热加载 | `nginx -s reload` 不中断现有连接 |
| 高可用切换 | 双机热备切换时间不超过 3 秒 |
| 运行稳定性 | 支持 7×24 小时持续压力运行，无 crash、内存泄漏、句柄泄漏 |

### 7.3 可维护性要求

- 日志格式、字段、分类与现网保持一致
- 支持查看请求统计、连接状态、Lua 内存及运行指标
- 提供健康检查接口供 LB 或平台探测
- 提供配置转换工具，支持 Java 配置向 OpenResty 配置迁移

| 项目 | 要求 |
| --- | --- |
| 部署模式 | 主路模式、旁路模式 |
| 工作模式 | 反向代理模式、NC 模式 |
| 硬件型号 | SG-5200/5300/5600/6000/8000 |
| 高可用 | 支持双机热备 |

### 8.4 迁移实施要求

| 迁移项 | 要求 |
| --- | --- |
| 配置文件转换 | 提供命令行或工具化转换能力，支持一键或半自动迁移 |
| 证书与私钥 | 格式保持兼容，可重组目录结构但不得影响使用 |
| 用户数据 | 数据库结构原则上不变，由 OpenResty 通过 Lua 或后端接口访问 |
| 灰度发布 | 支持新旧网关并存与部分流量导入 |
| 真实配置验证 | 至少使用 3 组不同规模客户配置进行迁移验收 |

## 10. 验收标准

### 10.1 功能验收

- AC01：原白皮书附录功能项逐项验证通过
- AC02：反向代理模式下 B/S 与 C/S 应用访问正常
- AC03：NC 模式各拓扑场景访问正常
- AC04：SSO 代填、证书透传、Cookie 跨域能力正常
- AC05：证书、口令、双因子、RADIUS、LDAP、AD 认证方式正常
- AC06：RBAC、动态角色匹配、访问策略行为一致
- AC07：终端安全检查、绑定、零痕迹访问能力正常
- AC08：审计日志字段、格式、分类与 Java 版本一致
- AC09：集中管控平台可正常纳管新版本网关

### 10.2 性能验收

- AC10：核心性能指标不低于所选硬件基线
- AC11：SSL 握手延迟达到目标或不劣于现网约束
- AC12：7×24 小时压力测试无 crash、无内存泄漏
- AC13：热加载执行 1000 次无异常
- AC14：双机热备切换时间不超过 3 秒

### 10.3 迁移验收

- AC15：现有客户配置文件可导入，或通过迁移工具完整转换
- AC16：至少 3 组真实客户配置迁移后应用、用户、策略、证书完整恢复
- AC17：迁移后访问行为与旧网关一致
- AC18：客户端无需升级即可接入新网关

## 11. 风险与应对

| 风险编号 | 风险 | 等级 | 影响 | 应对措施 |
| --- | --- | --- | --- | --- |
| R1 | 国密 OpenSSL 与 OpenResty 集成困难 | 🔴 高 | 国密功能可能无法按目标迁移 | Phase 0 必须先完成 PoC 验证 |
| R2 | SSO 表单代填 Lua 重写复杂度高 | 🟡 中 | SSO 功能完整性风险 | 分阶段迁移，优先支持主流场景 |
| R3 | 双机热备状态同步机制不成熟 | 🟡 中 | 高可用可靠性风险 | 采用 shared dict + 后端同步设计并专项压测 |
| R4 | Lua 第三方协议库成熟度不足 | 🟡 中 | 外部认证与日志链路受限 | 预留后端 API 转发备选方案 |
| R5 | NC 模式流控行为与原版差异 | 🟡 中 | 客户端兼容性与稳定性风险 | 必要时保留最小辅助进程承担部分控制或转发职责 |
| R6 | 性能未达预期 | 🟢 低 | 迁移收益不明显 | 保留 Java 版本回退方案并开展阶段性基准验证 |

## 12. 实施阶段建议

| 阶段 | 目标 | 主要产出 | 预估周期 |
| --- | --- | --- | --- |
| Phase 0 | 国密可行性验证 | 国密 OpenResty PoC 报告 | 1 周 |
| Phase 1 | 核心流量引擎迁移 | SSL/TLS、代理、认证、RBAC 实现 | 4-6 周 |
| Phase 2 | 高级能力迁移 | SSO、NC、双机热备实现 | 3-4 周 |
| Phase 3 | 审计监控适配 | 日志、监控、SNMP、Syslog 适配 | 2 周 |
| Phase 4 | 全量测试与性能验证 | 功能测试报告、性能测试报告 | 2-3 周 |
| Phase 5 | 兼容性验证与试运行 | 迁移验收报告、试运行报告 | 1-2 周 |

优先级建议：`Phase 0 > Phase 1 > Phase 4 > Phase 2 > Phase 3 > Phase 5`

> Phase 0 为硬阻塞项。如果国密能力无法在 OpenResty 下可靠落地，整体方案需要重新评估。

## 13. 结论

> TSG 7.6 架构迁移的核心是“换引擎不换功能”。本次迁移不是产品功能重做，而是对外服务引擎从 Java 向 OpenResty/Lua 的体系替换。项目成败的关键取决于国密兼容性、SSO 复杂场景、双机热备同步与客户端无感兼容四项能力是否按计划落地。
