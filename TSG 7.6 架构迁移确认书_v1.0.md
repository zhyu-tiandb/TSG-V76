## 2. 背景与目标

### 2.1 项目背景

TSG（trustMore 安全网关）当前版本的对外服务引擎基于 Java 实现，负责处理 SSL/TLS 终结、反向代理、认证鉴权、单点登录等核心业务流量。随着业务场景对高并发、低延迟的要求不断提升，Java 引擎在以下方面存在瓶颈：

- **并发处理能力**：Java 引擎在高并发 SSL 连接场景下 GC 压力大，响应延迟不稳定
- **资源开销**：JVM 内存占用高，同等硬件条件下有效吞吐受限
- **协议处理效率**：SSL 握手、HTTP 解析等 I/O 密集型操作在 Java 层的性能不如原生实现
- **运维复杂度**：JVM 调优门槛高，部署相对笨重

OpenResty（基于 Nginx + LuaJIT）天然具备高并发、低资源占用的特性，且原生支持 SSL/TLS 终结、HTTP 代理等核心能力，适合作为网关对外服务引擎。

### 2.2 迁移目标

| 目标 | 说明 | 量化标准 |
|------|------|---------|
| 功能等价 | 所有现有功能在 OpenResty 引擎下 100% 可用 | 0 功能缺失 |
| 性能提升 | 利用 OpenResty 提升并发和吞吐能力 | 并发连接数 ≥ Java 版本，SSL 握手延迟降低 ≥30% |
| 行为兼容 | 用户侧、管理侧无感知变化 | 现有配置可无缝导入，管理操作不变 |
| 国密兼容 | 国密 SSL/TLS 协议在 OpenResty 下完整支持 | SM2/SM3/SM4 算法 + 国密 SSL 协议全支持 |
| 部署兼容 | 硬件规格不变，部署方式不变 | SG-5200/5300/5600/6000 型号全部支持 |

### 2.3 非目标（Out of Scope）

- 不新增产品功能
- 不改变客户端交互逻辑
- 不改变管理页面 UI
- 不改变硬件平台
- 不改变部署方式（主路/旁路模式不变）

## 3. 迁移范围与模块影响分析

### 3.1 模块影响分类说明

| 分类 | 标记 | 含义 |
|------|------|------|
| 引擎迁移 | 🔄 | 涉及 Java → OpenResty 核心逻辑改造，功能行为不变 |
| 适配调整 | ⚠️ | 引擎切换后需适配或验证，逻辑可能有小调整 |
| 无变化 | ➡️ | 不涉及引擎层，后端/前端/硬件逻辑不变 |

### 4.1 总体分布

| 影响类型 | 数量 | 占比 | 说明 |
