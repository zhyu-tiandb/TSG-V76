## 2. 系统定位与边界

### 2.1 系统定位

TSG 7.6 是 trustMore 安全网关的一次**底层架构迁移**，将对外服务引擎从 Java 替换为 OpenResty（Nginx + LuaJIT），功能等价、性能提升、行为兼容。

### 2.3 改造边界

| 层级 | 改造范围 | 说明 |
|------|---------|------|
| **OpenResty 引擎层** | 全部重写 | Java → OpenResty/Lua，这是本次迁移的核心 |
| **后端服务层** | 接口适配 | 引擎层调用方式变化（Java 内部调用 → HTTP/Lua 调用），接口协议不变 |
| **管理面** | 不变 | Web UI、三权分立、配置管理保持原样 |
| **操作系统层** | 不变 | 网络栈、国密驱动、硬件驱动保持原样 |
| **客户端** | 不变 | 用户端逻辑不涉及 |

### 3.2 架构设计决策

| # | 决策点 | 决策 | 依据 |
|---|--------|------|------|
| D1 | SSL 终结放在哪层 | OpenResty Nginx 核心层（原生 SSL） | OpenResty 原生 SSL 性能远优于 Lua 实现 |
| D2 | 国密 SSL 怎么支持 | 国密 OpenSSL/Tongsuo 编译进 Nginx | 必须在 SSL 终结层支持国密算法 |
| D3 | 认证逻辑放哪层 | Lua 业务层（ssl_certificate_by_lua / access_by_lua） | 认证属于请求处理流程中的业务逻辑 |
| D4 | 第三方认证（RADIUS/LDAP）怎么处理 | Lua 直连或转发后端（Phase 0 评估） | 取决于 Lua 协议库成熟度 |
| D5 | 会话状态怎么管理 | OpenResty shared_dict + 后端持久化 | shared_dict 满足高性能读取，后端满足持久化和热备同步 |
| D6 | 配置管理怎么交互 | 管理面写后端 → 后端推送/拉取到 OpenResty | 管理面不直接操作 Nginx 配置文件 |
| D7 | NC 模式怎么实现 | OpenResty stream 模块 + Lua | stream 模块原生支持 TCP/UDP 代理 |
| D8 | SSO 表单代填怎么实现 | Lua body_filter 解析 HTML + 注入表单 | 必须在响应体过滤层实现 |

### 4.2 模块职责说明

#### 引擎层模块

| 模块 | 职责 | 对应白皮书能力 | OpenResty 实现方式 |
|------|------|--------------|-------------------|
| M1: SSL/TLS 终结 | 处理所有入站 SSL/TLS 连接，包括国密 SSL | 链路安全（2.1） | ngx.ssl + 国密 OpenSSL |
| M2: 反向代理 | 将请求转发到后端应用，支持 B/S 和 C/S | 应用保护（2.6.1） | proxy_pass + stream |
| M3: 认证鉴权 | 处理用户身份认证（证书/口令/双因子） | 用户认证（2.2） | ssl_certificate_by_lua + access_by_lua |
| M4: RBAC 策略执行 | 根据用户角色和策略控制访问权限 | 权限管理（2.4） | access_by_lua |
| M5: SSO 单点登录 | 表单代填、证书透传、Cookie 跨域 | 单点登录（2.8） | body_filter_by_lua + header_filter_by_lua |
| M6: NC 隧道 | TCP/UDP 隧道建立与数据转发 | NC 模式（4.2） | stream + Lua |
| M7: 统一门户 | 门户页面服务、应用列表展示 | 统一门户（2.9） | Lua 模板渲染 |
| M8: 终端安全校验 | 终端指纹验证、安全基线检查 | 终端安全（2.5） | access_by_lua + 后端 API |
| M9: 日志采集 | 记录用户访问日志，转发 SysLog | 安全审计（2.11） | log_by_lua |
| M10: 会话管理 | 在线用户状态、超时注销、拔 KEY 注销 | 在线生命周期（2.2.2.4） | shared_dict + 后端同步 |

## 6. 核心业务流程

### 6.1 用户访问主流程（反向代理模式）

```
1) 用户发起 HTTPS 请求
2) M1(SSL终结): 建立 SSL 连接，加载服务器证书
3) M1(SSL终结): 如果双向认证，验证客户端证书（调用 M13 CRL/OCSP）
4) M3(认证鉴权): 检查用户登录状态（查询 M10 会话）
   ├─ 未登录 → 跳转登录页面（M7 统一门户）
   │   ├─ 证书登录 → ssl_certificate_by_lua 验证
   │   ├─ 口令登录 → access_by_lua + 后端 API 验证
   │   └─ 第三方认证 → 转发 M15
   └─ 已登录 → 继续
5) M4(RBAC策略): 检查用户是否有权访问目标应用
   ├─ 无权 → 返回 403
   └─ 有权 → 继续
6) M8(终端安全): 检查终端是否可信（绑定、安全基线）
   ├─ 不可信 → 拒绝
   └─ 可信 → 继续
7) M2(反向代理): 转发请求到后端应用
8) M5(SSO): 如需要，注入认证信息（表单代填/证书透传）
9) M9(日志采集): 记录本次访问日志
10) 返回响应给用户
```

### 6.2 NC 模式隧道建立流程

```
1) 客户端发起 TCP/UDP 连接
2) M1(SSL终结): 建立 SSL 连接
3) M3(认证鉴权): 验证客户端身份
4) M4(RBAC策略): 检查访问策略
5) M6(NC隧道): 建立 stream 隧道到对端网关/内部服务
6) M9(日志采集): 记录隧道建立日志
7) 双向数据转发
```

### 6.3 管理面配置下发流程

```
1) 管理员通过 M17(Web UI) 修改策略
2) M17 调用 M12(策略管理) 后端 API
3) M12 持久化策略数据
4) M12 推送策略变更到 OpenResty shared_dict
5) OpenResty reload 或 Lua 热加载生效
6) M16(高可用) 同步到备节点
```

## 7. 关键对象关系模型

### 7.1 核心实体关系

```
用户 ──1:N──→ 用户账号（主账号 + 从账号）
用户 ──N:M──→ 角色
角色 ──N:M──→ 应用资源
用户 ──N:M──→ 终端设备
应用资源 ──1:N──→ 访问策略（时间/IP/端口规则）
用户 ──1:1──→ 会话（在线状态）
终端 ──1:1──→ 终端指纹（硬件信息）
用户 ──1:N──→ 数字证书
```

### 7.2 策略执行链

```
访问策略 = 用户角色 + 应用资源 + 时间规则 + IP规则 + 端口规则 + 终端安全策略

请求 → 匹配角色 → 匹配应用 → 匹配时间 → 匹配IP → 匹配终端 → 放行/拒绝
```

## 9. 非功能设计

### 9.1 性能设计

| 设计要点 | 实现方式 |
|---------|---------|
| SSL 卸载 | OpenResty 原生 SSL 终结，减少 Lua 层介入 |
| 连接复用 | keepalive + upstream 连接池 |
| 缓存策略 | shared_dict 缓存策略数据，减少后端查询 |
| 压缩 | gzip 压缩（Nginx 原生），自动跳过已压缩格式 |

### 9.2 高可用设计

| 设计要点 | 实现方式 |
|---------|---------|
| 主备切换 | VRRP 或后端心跳检测，切换时间 ≤ 3 秒 |
| 会话同步 | shared_dict 定期同步到备节点 |
| 配置同步 | 后端数据库主从复制 + Nginx 配置推送 |
| 不中断 reload | `nginx -s reload` 不中断已有连接 |

### 9.3 安全设计

| 设计要点 | 实现方式 |
|---------|---------|
| 引擎进程隔离 | Nginx worker 进程隔离，单 worker 崩溃不影响全局 |
| Lua 代码安全 | LuaJIT 沙箱，限制文件系统和网络访问 |
| 敏感数据保护 | 私钥不进入 shared_dict，仅 worker 内持有 |
| 日志防篡改 | 日志写入后端存储，OpenResty 层不保留原始日志 |

## 10. 设计取舍说明

| # | 取舍 | 选择 | 原因 |
|---|------|------|------|
| 1 | 认证逻辑全放 Lua vs 部分转发后端 | 核心认证在 Lua，第三方认证评估后决定 | 减少 RTT，但第三方协议库成熟度待验证 |
| 2 | shared_dict 存会话 vs 全部存后端 | shared_dict + 后端双写 | shared_dict 满足性能，后端满足持久化 |
| 3 | 配置热加载方式 | shared_dict 缓存 + 定期拉取 | 避免直接操作 Nginx 配置文件，降低出错风险 |
| 4 | SSO 表单代填范围 | Phase 1 支持主流方式，Phase 2 全量 | 降低首期复杂度，优先覆盖 80% 场景 |
