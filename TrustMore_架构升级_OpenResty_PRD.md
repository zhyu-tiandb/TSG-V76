# 需求文档：TrustMore 安全网关架构升级（Java → OpenResty）

| 文档版本 | 1.0 | 拟制日期 | 2026-05-07 |
|---------|-----|---------|-------------|
| 项目名称 | TrustMore 安全网关架构升级 | 评审级别 | 内部评审 |
| 变更类型 | 架构重构（对外服务引擎） | 功能变更 | 无 |

---

## 1. 背景与目标

### 1.1 背景
当前 TrustMore 安全网关的对外服务引擎基于 Java 技术栈实现（含 SSL 处理、反向代理、NC 模式核心、单点登录代填、统一门户等）。随着客户对高并发、低延迟、轻量部署的要求不断提高，Java 引擎在以下方面存在瓶颈：

- 内存占用较高，尤其在大量 SSL 连接场景下；
- 并发处理能力受限于 JVM 线程模型；
- 部署和运维相对复杂，调优门槛高。

OpenResty（Nginx + Lua）在高性能网络 I/O、SSL 卸载、动态配置、资源占用方面具有明显优势，且已在众多安全网关类产品中得到验证。

### 1.2 目标
- **功能完全等价**：所有现有功能（见附录 A）行为与原 Java 版本一致，用户和管理员无感知。
- **性能提升**：在相同硬件条件下，新建连接速率、加密带宽、并发连接数不低于原版本。
- **平滑迁移**：支持现有配置、证书、用户数据无缝迁移至新架构。
- **降低运维成本**：减少内存占用，支持热加载配置，简化部署。

## 2. 范围

### 2.1 改造范围
| 模块 | 说明 |
|------|------|
| HTTP/HTTPS 协议处理 | 替换为 OpenResty 的 http/ssl 模块，保留国密 SSL 支持 |
| 反向代理模式 | 使用 Nginx 反向代理 + Lua 扩展实现原有路由、URL 重写、Cookie 注入等逻辑 |
| NC 模式（TCP/UDP 隧道） | 保留隧道建立和控制逻辑，由 OpenResty 负责握手、策略校验，隧道数据转发可使用 stream 模块或外部进程 |
| 单点登录代填 | 使用 Lua 解析响应页面、注入表单或 header，行为与原版一致 |
| 统一门户 | 用 Lua + 模板渲染实现原门户功能和授权应用列表展示 |
| 终端安全策略校验 | OpenResty 负责接收客户端上报的终端信息，执行策略判断并返回结果 |
| 审计日志 | 保持原日志格式和字段，通过 Lua 记录并支持 SYSLOG 转发 |

### 2.2 不变范围
- 管理后台（若与管理引擎分离且不涉及对外服务，则不改造）
- 用户/角色/应用数据库结构
- 客户端插件（Windows / Linux 插件无需升级）
- 硬件适配层（如加密卡、随机数发生器）
- 资质认证相关的密码模块调用接口

## 3. 功能等价性要求

原白皮书附录 A 中的所有功能条目必须完整保留，此处仅列出高风险或易遗漏项，评审时需重点确认：

| 功能点 | 原实现方式 | OpenResty 实现要求 |
|--------|------------|---------------------|
| 国密 SSL（GM/T 0024） | Java JCE + 国密库 | 使用 Nginx 国密补丁或 TongSuo 版本，支持 SM2/SM3/SM4 套件 |
| 证书有效性验证（CRL/OCSP） | Java 定时拉取+缓存 | Lua 实现定时下载 CRL 并缓存，或调用外部脚本 |
| 动态端口应用（FTP） | Java 解析控制流动态开放端口 | 需保留原逻辑，可由 OpenResty 调用 Lua 脚本或转发至辅助服务 |
| 用户终端绑定（多对多） | Java 内存/数据库检查 | Lua 访问数据库或共享缓存，策略判断逻辑完全一致 |
| 零痕迹访问（退出后清除缓存） | Java 控制客户端清除 | 由客户端执行，网关只需发送清除指令（协议不变） |
| 三权分立管理 | 管理端控制 | 不涉及，管理端若统一则需检查是否依赖 Java 接口，可保留原接口不变 |

**特别注意**：NC 模式下的隧道协议必须与现有客户端完全兼容，避免升级客户端。

## 4. 非功能性需求

### 4.1 性能指标（以 SG-6000 硬件为参考）

| 指标 | 原 Java 版本 | OpenResty 目标值 | 备注 |
|------|-------------|------------------|------|
| 国密新建连接数（次/秒） | 4500 | ≥ 4500 | 同等硬件 |
| 国密 SSL 事务处理速率（次/秒） | 8000 | ≥ 8000 | |
| 国密最大并发连接数 | 10 万 | ≥ 10 万 | |
| 国密加密带宽（Mbps） | 900 | ≥ 900 | |
| HTTP 请求处理延迟（P99） | - | ≤ 原版本 120% | 不劣化 |
| SSL 握手延迟（P99） | - | 不劣化 | |

### 4.2 资源占用

| 项目 | 要求 |
|------|------|
| 内存占用（空闲时） | ≤ Java 版本 **70%** |
| CPU 使用率（同等压力） | 不高于 Java 版本 |
| 磁盘 I/O（日志写入） | 与原版本持平 |

### 4.3 高可用

- **双机热备**：支持主备模式，配置和会话同步（会话同步如不需要可降级为仅配置同步）。
- **配置热加载**：修改策略后执行 `nginx -s reload` 不中断现有连接。
- **健康检查**：提供专用接口供负载均衡器探测。

### 4.4 可维护性

- 日志格式完全兼容，现有日志分析工具可直接使用。
- 支持实时查看 OpenResty 的请求统计、连接状态、Lua 内存使用。
- 提供配置转换工具（Java XML/JSON → Nginx conf + Lua 脚本）。

## 5. 迁移与兼容性要求

| 迁移项 | 要求 |
|--------|------|
| 配置文件转换 | 提供命令行工具，将原 Java 版网关配置一键转换为 OpenResty 格式 |
| 证书与私钥 | 直接拷贝，格式（PEM/DER）不变，目录结构可重新组织 |
| 用户数据 | 数据库无需变更，OpenResty 通过 Lua 连接原有数据库 |
| 客户端升级 | **不要求**，需验证现有客户端（ActiveX / exe / Linux 客户端）可与新网关正常交互 |
| 灰度发布能力 | 支持将部分流量导入新网关，与旧网关并存 |

---

## 6. 约束条件

- 不得引入新的第三方商业软件或要求额外授权。
- 原有国密硬件（如 USB Key、加密卡）的调用方式不得改变。
- 若 OpenResty 原生无法实现的极少数功能（如复杂动态端口协议），允许保留最小 Java 辅助进程，但必须明确评估和文档化。

## 7. 验收标准

### 7.1 功能验收
执行原白皮书附录 A 中所有功能测试用例（共约 150+ 项），结果必须与 Java 版本完全一致。

### 7.2 性能验收
在相同硬件（SG-5600 / SG-6000）上，使用相同的测试工具（如 LoadRunner、Scapy + 国密 SSL 脚本）运行混合场景，各项性能指标不低于基线。

### 7.3 稳定性验收
- 7×24 小时压力测试（逐步加压至目标并发 120%），无 crash、内存泄漏、句柄泄漏。
- 支持热加载配置 1000 次，无异常。

### 7.4 迁移验收
使用真实客户配置文件（至少 3 个不同配置规模）导入新网关，所有应用、用户、策略、证书完整恢复，且访问行为与旧网关一致。

## 8. 风险与应对

| 风险 | 概率 | 影响 | 应对措施 |
|------|------|------|----------|
| Lua 实现国密 SSL 性能不足 | 中 | 高 | 采用 Nginx 国密发行版（如 TongSuo）在 C 层完成，Lua 仅做配置 |
| NC 模式 TCP 隧道在 OpenResty 中流控与原版不一致 | 中 | 中 | 保留原隧道转发逻辑的 C 模块或独立进程，OpenResty 仅做控制面 |
| CRL/OCSP 实时检查影响并发 | 低 | 中 | 引入共享缓存（lua-resty-lrucache）并异步更新 |
| 配置文件转换工具开发成本 | 低 | 低 | 可先实现半自动脚本，后续迭代完善 |

## 9. 附录：受影响的功能清单（与原白皮书完全一致）

> 以下功能必须 100% 保留，仅底层实现更改。

- 数据传输加密（国密/国际 SSL）
- 双网隔离

- 应用/角色/用户/终端管理
- HTTP 压缩、高并发支持
- 双机热备、第三方负载均衡
- 单点登录（代填+证书透传）
- 统一门户
- 三权分立、管理员安全策略
- 审计日志（本地 + SYSLOG）
- 实时监控（CPU/内存/连接数等）
