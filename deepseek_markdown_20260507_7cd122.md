## 1. 背景与目标

### 1.1 项目背景

TSG（TrustMore 安全网关）当前版本（7.x）的对外服务引擎基于 **Java** 技术栈实现，负责处理 SSL/TLS 终结、反向代理、认证鉴权、单点登录、NC 模式隧道等核心业务流量。随着客户对高并发、低延迟、轻量部署的要求不断提升，Java 引擎在以下方面暴露瓶颈：

- **并发处理能力**：高并发 SSL 连接场景下 JVM GC 压力大，响应延迟不稳定
- **资源开销**：JVM 内存占用高，同等硬件条件下有效吞吐受限
- **协议处理效率**：SSL 握手、HTTP 解析等 I/O 密集型操作在 Java 层的性能不如原生实现
- **运维复杂度**：JVM 调优门槛高，部署相对笨重

OpenResty（Nginx + LuaJIT）天然具备高并发、低资源占用的特性，且原生支持 SSL/TLS 终结、HTTP 代理、Stream 代理等核心能力，已在众多安全网关类产品中得到验证。

### 1.2 迁移目标

| 目标 | 说明 | 量化标准 |
|------|------|---------|
| **功能等价** | 所有现有功能在 OpenResty 引擎下 100% 可用 | 0 功能缺失，白皮书附录 A 全量通过 |
| **性能提升** | 利用 OpenResty 提升并发和吞吐能力 | 并发连接数 ≥ Java 版本，SSL 握手延迟降低 ≥30% |
| **行为兼容** | 用户侧、管理侧无感知变化 | 现有配置可无缝导入，管理操作不变 |
| **国密兼容** | 国密 SSL/TLS 协议在 OpenResty 下完整支持 | SM2/SM3/SM4 算法 + 国密 SSL 协议全支持 |
| **部署兼容** | 硬件规格不变，部署方式不变 | SG-5200/5300/5600/6000 型号全部支持 |

### 1.3 非目标（Out of Scope）

- 不新增产品功能
- 不改变客户端交互逻辑
- 不改变管理页面 UI（如管理端与引擎分离）
- 不改变硬件平台
- 不改变部署方式（主路/旁路模式不变）

## 2. 范围与模块影响分析

### 2.1 改造范围总览

**核心原则**：仅替换对外服务引擎，所有管理面、后端数据库、客户端逻辑保持不变。

| 改造内容 | 说明 |
|----------|------|
| HTTP/HTTPS 协议处理 | 替换为 OpenResty 的 http/ssl 模块，保留国密 SSL 支持 |
| 反向代理模式 | 使用 Nginx 反向代理 + Lua 扩展实现原有路由、URL 重写、Cookie 注入 |
| NC 模式（TCP/UDP 隧道） | 使用 OpenResty stream 模块 + Lua 实现隧道建立、策略校验、数据转发 |
| 单点登录代填 | 使用 Lua 解析响应页面、注入表单或 header，行为与原版一致 |
| 统一门户 | 用 Lua + 模板渲染实现原门户功能和授权应用列表展示 |
| 终端安全策略校验 | OpenResty 负责接收客户端上报的终端信息，执行策略判断并返回结果 |
| 审计日志 | 保持原日志格式和字段，通过 Lua 记录并支持 SYSLOG 转发 |

## 3. 性能与非功能性需求

### 3.1 性能指标

**现有基线（Java 引擎，SG-5300 为例）**：

| 指标 | 当前值 |
|------|-------|
| 国密最大新建连接数 | 3,000 次/秒 |
| 国密 SSL 事务处理速率 | 5,000 次/秒 |
| 国密最大并发连接数 | ≤30,000 |
| 国密加密带宽吞吐量 | 850 Mbps |
| 接入用户数 | ≤10,000 |

**OpenResty 迁移目标**：

| 指标 | 目标值 | 提升依据 |
|------|--------|----------|
| 国密最大新建连接数 | ≥4,500 次/秒 | OpenResty 事件驱动 + 无 GC 停顿 |
| 国密 SSL 事务处理速率 | ≥8,000 次/秒 | 原生 SSL 终结 + OpenSSL 硬件加速 |
| 国密最大并发连接数 | ≥50,000 | Nginx 事件模型天然优势 |
| 国密加密带宽吞吐量 | ≥1,200 Mbps | 减少内存拷贝和 JVM 开销 |
| 接入用户数 | ≥15,000 | 内存占用降低，可容纳更多会话 |
| SSL 握手 P99 延迟 | ≤ 原版本 70% | 原生 SSL 栈效率更高 |

> 注：上述目标值为预期参考，需在开发完成后实测确认。

### 3.2 资源占用要求

| 项目 | 要求 |
|------|------|
| 内存占用（空闲时） | ≤ Java 版本 **70%** |
| CPU 使用率（同等压力） | 不高于 Java 版本 |
| 磁盘 I/O（日志写入） | 与原版本持平 |

### 3.3 高可用要求

- **双机热备**：支持主备模式，切换时间 ≤ 3 秒
- **配置热加载**：修改策略后 `nginx -s reload` 不中断现有连接
- **健康检查**：提供专用接口供负载均衡器探测

### 3.4 可维护性要求

- 日志格式完全兼容，现有日志分析工具可直接使用
- 支持实时查看 OpenResty 的请求统计、连接状态、Lua 内存使用
- 提供配置转换工具（Java XML/JSON → Nginx conf + Lua 脚本）

| R3 | 双机热备状态同步机制不成熟 | 🟡 中 | 高可用不可靠 | 利用 OpenResty shared dict + 后端同步 |
| R4 | Lua 第三方库不成熟（RADIUS/LDAP） | 🟡 中 | 第三方认证功能受限 | 评估后端 API 转发方案作为备选 |
| R5 | 性能未达预期 | 🟢 低 | 迁移收益不明显 | 保留 Java 版本作为回退方案 |

## 8. 实施计划与阶段划分

| 阶段 | 目标 | 产出 | 预估周期 |
|------|------|------|----------|
| **Phase 0** | 国密 PoC 验证 | 国密 SSL + OpenResty 可行性报告 | 1 周 |
| **Phase 1** | 核心流量引擎迁移 | SSL/TLS 终结 + 反向代理 + 认证 + RBAC | 4-6 周 |
| **Phase 2** | 高级功能迁移 | SSO + NC 模式 + 双机热备 | 3-4 周 |
| **Phase 3** | 监控审计适配 | 日志 + 监控 + SNMP + SysLog | 2 周 |
| **Phase 4** | 全量功能测试 + 性能测试 | 测试报告 + 性能报告 | 2-3 周 |
| **Phase 5** | 兼容性验证 + 试运行 | 验收报告 | 1-2 周 |

**优先级建议**：Phase 0（阻塞项）> Phase 1（核心）> Phase 4（验证）> Phase 2 > Phase 3 > Phase 5

> **Phase 0 是硬阻塞项**，如果国密 OpenSSL 无法与 OpenResty 正常集成，整个迁移方案需要重新评估。

## 10. 一句话结论
